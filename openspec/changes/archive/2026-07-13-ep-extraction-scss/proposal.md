# Proposal: ep-extraction-scss

> SCSS phase of the element-plus-free DS initiative, anchored on `config-provider`.
> Artifact store: hybrid. Mirror of engram `sdd/ep-extraction-scss/proposal`.

## Intent

### Problem

The design system's SCSS layer does not own its own styling foundation. Every
`--gui-*` design token, every BEM selector, and the `gui` namespace itself are
generated by importing element-plus internals from
`element-plus/theme-chalk/src/*` at build time. 53 `.scss` files across the DS
`@use` element-plus theme-chalk directly, in two shapes:

1. **~48 shared mixin/token imports** — `mixins/mixins`, `mixins/utils`,
   `mixins/var` (`_var.scss`), `common/var`, plus `common/transition` and
   `common/popup`.
2. **~20 full per-component stylesheet imports** — files that re-export EP's
   compiled component CSS wholesale (`badge.scss`, `menu*.scss`, `popover.scss`,
   `form-item.scss`, `skeleton*.scss`, `radio*.scss`, `dropdown.scss`, etc.).

The linchpin is `components/config-provider/config.styles.scss`, which
`@forward`s EP's `mixins/config` (namespace) and `common/var` (brand colors),
then `@use`s `base.scss` to emit the entire `:root { --gui-* }` token block.
config-provider is also a JS render-island: `ConfigProvider.vue` renders the real
`<el-config-provider namespace="gui">` from element-plus to propagate `gui` to
every descendant's `useNamespace()` at runtime. Both the SCSS bootstrap and the
JS runtime currently borrow their `gui` value from element-plus internals.

### Why now

This is the SCSS phase of a broader initiative to make the DS element-plus-free.
The styling layer is the foundation the JS render-island migrations sit on: token
generation and the `gui` namespace must be DS-owned before the remaining islands
can be replaced. config-provider is the single point where the SCSS bootstrap and
a JS island are inseparable, so it is the correct anchor to migrate first and
validate the whole token pipeline end-to-end. Doing it now also removes an active
fragility: the namespace only resolves to `gui` (not `el`) by relying on Sass
module singleton semantics plus an unverified cross-repo Sass aggregation order in
the consumer apps.

### Success looks like

- The `--gui-*` tokens and BEM styling are generated by DS-owned Sass in
  `common/g-utils/styles/`, no longer importing `element-plus/theme-chalk` for the
  migrated surface.
- Generated CSS is **byte-exact** vs. today's output (selectors, `--gui-*` custom
  properties, and values identical) — proven by a parity harness built in this
  change, since CI has no stylelint/CSS-snapshot gate.
- The `gui` namespace remains configurable from ONE place
  (`common/g-utils/styles/config.scss` `$namespace`); the migration strengthens
  the single-source design rather than forking it.
- config-provider's SCSS bootstrap and its JS render-island are both DS-native and
  source `gui` from the single point of change.
- No visual change, no public API change (props/emits/slots) for any component.

## Scope

### In scope

- **Port the 4 missing shared theme modules** into `common/g-utils/styles/`:
  `mixins/utils`, `mixins/var` (`_var`), `common/var`, and `base` (the `:root`
  token emitter). Byte-exact ports of element-plus source.
- **config-provider SCSS + JS migration (together)** — the one island whose SCSS
  and JS are inseparable:
  - Rebuild `config.styles.scss` to bootstrap `--gui-*` from DS-owned modules
    (no element-plus import).
  - Replace the `ElConfigProvider` JS render-island in `ConfigProvider.vue` with a
    DS-native config-provider that sources `gui` from `g-utils/config`, not a
    hardcoded template literal.
- **Byte-exact CSS parity harness** — build tooling that compiles the generated
  CSS before and after and diffs it, as a deliverable of this change. This is the
  safety net that CI does not provide today.
- **Fix the `badge.styles.scss` hardcoded `.gui-badge__content` selector** — bring
  it under the mixin/namespace system so the "never hardcode `gui`" invariant holds.
- **Repoint the coupled `.scss` files off element-plus** where safe — the ~48
  shared-mixin files to consume DS-owned modules, and the full-component
  stylesheet files that are NOT entangled with an unmigrated JS render-island.
- **Verify the cross-repo namespace-aggregation assumption** as an early task
  (before design locks an approach): inspect consumer-side Sass import order in
  front-b2b / front-admin or a build artifact to confirm legacy files resolve to
  `gui` and not `el` in production.

### Out of scope (non-goals)

- **Reimplementing the JS of the other 7 render-islands** — badge, menu, popover,
  radio-group, form-item, skeleton keep their current JS wrappers (still rendering
  element-plus DOM). Only their SCSS is repointed where the DOM contract is stable;
  their JS migration is a future phase.
- **Dropping the element-plus dependency entirely** — element-plus remains a
  required peer dependency until ALL JS render-islands (including the 7 above) are
  migrated. The SCSS phase alone cannot remove it.
- **Any visual or API change** — no new tokens, no selector changes, no
  props/emits/slots changes. Parity is the contract.
- **Full-component stylesheet files entangled with unmigrated JS islands** — see
  the sequencing decision below.

### Sequencing decision for the entangled full-component files

Of the 8 JS render-islands, 7 also own full-component SCSS that matches a DOM
still rendered by element-plus (badge, menu, popover, radio-group, form-item,
skeleton, config-provider). For these, the ported SCSS would have to keep matching
a DOM the DS does not yet control. The proposal treats them as follows:

- **config-provider**: migrate SCSS + JS together in this change (locked decision).
- **The other 6 entangled islands**: repoint only the **shared-mixin dependency**
  (so they consume DS-owned tokens/mixins instead of EP's) when that can be done
  byte-exact. **Defer** their **full-component stylesheet** repoint until their JS
  render-island migration lands, so the CSS and the DOM it targets move together.
- **Non-island full-component files** (e.g. dropdown, scrollbar, table, collapse,
  drawer, tag, progress, segmented, select-v2, etc.): repoint in later slices,
  each byte-diffed through the parity harness.

## Approach

Adopt exploration **Approach 1 — bootstrap-first vertical slice, then fan out.**
Validate the riskiest, most load-bearing piece (token + namespace generation) in
isolation on config-provider with a real before/after CSS comparison, before
repointing dozens of dependent files that all rely on it being correct.

### Rationale

- De-risks the central piece first: a working, byte-verified token pipeline exists
  before 50+ files are touched. A subtle namespace/token bug is caught in one place
  instead of every file at once.
- Produces the chained/stacked-PR shape this repo already uses, each slice within
  the review budget and independently shippable.
- Surfaces the JS/SCSS coupling of the 7 overlapping islands early, so SCSS work is
  sequenced against the JS-island backlog and DOM/CSS never drift apart.

### Delivery shape (chained / stacked PRs, work-unit commits)

- **Slice 1** — g-utils token modules (`mixins/utils`, `mixins/var`, `common/var`,
  `base`) + config-provider SCSS bootstrap + config-provider JS replacement +
  the CSS parity harness + the cross-repo aggregation verification task.
- **Slice 2+** — repoint the ~48 shared-mixin files in logical component groups,
  each byte-diffed.
- **Final slices** — the non-entangled full-component stylesheet files; the
  entangled ones deferred to their JS-island migration.

### Guardrails carried through every slice

- element-plus is source of truth; every port is byte-exact (parity harness gates
  each slice).
- Single-source namespace: `gui` stays configurable only from
  `g-utils/styles/config.scss` `$namespace`; never hardcoded per component.
- No `any` in any TS touched (config-provider JS).
- New code comments / JSDoc in Spanish (neutral/professional), per project
  convention — the one exception to the English-artifact rule.
- Strict TDD active (`yarn test:run`); full Vitest suite on pre-push; CI `verify`
  lints only PR-changed files with `--max-warnings 0`.

## Open questions / risks carried forward

- The cross-repo Sass aggregation order in front-b2b / front-admin is unverified
  and the whole legacy-file port rests on it — must be confirmed as an early Slice 1
  task before design locks the port mechanics.
- Parity is unenforced by CI; the harness this change builds is the only gate, so
  its correctness is itself load-bearing.
- Sequencing of the 6 deferred entangled islands depends on the JS-island backlog
  cadence, which is outside this change's control.
